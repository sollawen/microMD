## 核心架构决策：探照灯模型

### 与 PRD-V2 第三章的根本差异

PRD 第三章描述的是**"双模式模型"**：

```
渲染模式（默认）：光标隐藏，方向键=滚动，按任意键 → 编辑模式
编辑模式：光标可见，方向键=移动光标，10 秒无操作 → 渲染模式
```

本方案采用更简单的**"探照灯模型"**：

```
始终处于同一状态：
  - 光标可见
  - 方向键移动光标（Micro 原生行为）
  - 所有 Micro 快捷键照常工作
  - 光标所在行的 segment 自动回退原生显示
  - 光标移开 → segment 自动恢复渲染显示
```

光标像一束探照灯，照到哪里哪里就回退原生、暴露 markdown 原文；光照不到的地方都是漂亮的渲染效果。

### 为什么选探照灯模型

1. **消灭项目最大风险源**。PRD 的双模式需要在 Micro 输入子系统前加拦截器、维护模式状态机、隐藏光标、处理 `Ctrl+S` / `:e` / VSplit 等大量边界情况。探照灯模型完全不需要这些。
2. **架构正确性 day-1 验证**。pipeline 从第一天就处理 100% 的行，不会拖到 v0.8 才发现接口设计缺陷。
3. **后续开发全部变成平行任务**。每个 renderer 升级独立、可并行、出 bug 影响范围小。
4. **UX 代价小**。用户在 terminal 看 Markdown 时本来就有光标，"探照灯"反而符合直觉。
5. **PRD 的"纯阅读模式"不丢弃**，作为可选 UX 增强放到 v0.6 加上去（10 秒计时器天然触发它）。

### 全文 100% 接管策略（自 v0.1 起）

pipeline 永远处理所有可见行。如果一行没有合适的"真"renderer，就用 stub：

- **`paragraph` segment（兜底）**：`Render()` 直接输出 `buffer.LineBytes(line)`，不做任何改造。光标在它范围内 → 回退原生（这总是成立，因为光标一定在某行）。
- **`inline` 渲染器（v0.1 阶段）**：passthrough，输入啥输出啥。

stub 的意义不是给用户看（用户看不到它们工作，因为光标所在行总是回退原生），而是**保证 pipeline 闭环、架构 day-1 完整**。

## v0.1 — 架构 + 探照灯 + 100% 接管（stub）

> **全项目的灵魂版本**。这一版做完，项目架构风险消除一半。

**目标**：渲染管线从第一天就接管所有 `.md` 文件的全部可见行。用户视角看到的是"光标走到哪段，那段就回退 markdown 原文；光标移开，那段变成 segment 输出（虽然 v0.1 阶段大部分 segment 输出 == 原文）"。

**范围**：
- 新建 `internal/display/md/` 子目录
- `pipeline.go`：管线主循环
  - 遍历可见行
  - 当前行的 segment 处于 edit state（光标命中）→ 委托 Micro 原生显示
  - 否则 → segment.Render() 输出
- `segment.go`：Segment 接口
  - `Contains(line int) bool`
  - `Render(line int) []RenderCell`（v0.1 先用简单的 byte + style 表示）
  - `IsEditState(cursors []*Cursor) bool`
- `paragraph.go`：**兜底 segment**，输出原始字节，不做任何改造
- `inline.go`：v0.1 阶段为 passthrough
- `table.go`：表格 segment（复用现有 `DetectTables` / `calcColWidths`）
- `bufwindow.go` 改造：在逐行渲染前先问 pipeline
- `.md` 文件不显示行号（gutter 移除）
- 配置开关 `mdrender`（默认 true）；关闭 → 完全回到 Micro 原生
- 旧 `markdown_table.go` 的 inline padding 路径**删除**（架构已替代它）

**不做**：
- 不做状态栏指示（v0.2 做）
- 不做 inline 真渲染（v0.3 做）
- 不做鼠标点击的额外处理（v0.1 的 segment 是 1:1 输出，鼠标行为天然正确）

**Demo**：
- 打开任意 `.md` 文件
- 光标移到表格里 → 表格回退 markdown 原文
- 光标移出表格 → 表格变对齐渲染
- 光标移到非表格行 → 看到原文（因为 paragraph stub 输出 == 原文）
- 行号消失

**验收标准**：
- 任意 `.md` 文件，pipeline 处理 100% 可见行（无任何 fallback 到 Micro 原生显示路径之外的代码）
- `mdrender: false` 完全回到 Micro 原生
- 非 `.md` 文件不受任何影响
- 现有 `markdown_table_test.go` 全部通过

## v0.2 — 表格渲染器（真）+ 状态栏

**目标**：让"探照灯"效果真正可见——表格 segment 输出和 paragraph stub 的原文输出有显著视觉差异。同时引入状态指示。

**范围**：
- 升级 `table.go`：
  - 单元格内容去除多余空格、统一 padding
  - 可选 ASCII 边框（`mdtableborder` 配置项）
  - 表头加粗（视终端能力）
- 状态栏 READING / EDITING 指示（`mdmodeindicator` 配置项）
  - 光标在任一 segment 内 → EDITING
  - 否则 → READING
- `.md` 文件状态栏显示（v0.1 已经移除了行号，这里补状态栏侧）

**Demo**：
- 打开含表格的 `.md`，光标在表格之外 → 看到漂亮表格
- 光标移入表格 → 表格变回原生 markdown，状态栏 EDITING
- 光标移出 → 表格恢复渲染，状态栏 READING

**验收标准**：
- 多光标场景行为正确（任一光标命中 segment 即 EDITING）
- 非 `.md` 文件状态栏不变

## v0.3 — inline 渲染器（粗体 / 斜体）+ 高亮叠加

**目标**：引入第二个能力维度——行内渲染。这是首个"真"renderer 升级 stub 的版本，验证"renderer 升级"流程。

**范围**：
- `inline.go` 升级：
  - 识别 `**` / `__` → 粗体
  - 识别 `*` / `_` → 斜体
  - 识别 `` ` `` → 行内代码（特殊样式）
  - 跨行闭合保守处理：标记未闭合则该行 passthrough
  - 输出：`[]byte`（去标记后的字符） + `[]tcell.Style`（每个字符的终端属性）
- **叠加 Micro 语法高亮颜色**：
  - 每个字符的最终样式 = Micro 高亮颜色 + inline 终端属性（粗体/斜体）
- 所有 segment 在输出内容字符时调用 inline 渲染器（表格单元格、paragraph 都用）
- 配置项 `mdbolditalic`（默认 true）

**Demo**：表格里的 `**重要**` 显示为粗体无 `**`；普通段落里的 `*斜体*` 同理。

**验收标准**：
- 不闭合的 `**` 不引起渲染错乱（该行 passthrough）
- inline 输出与 Micro 高亮叠加正确（颜色保留，粗体/斜体属性新增）

## v0.4 — 主要结构渲染器

**目标**：覆盖最常见的 Markdown 元素。**几乎没有新架构**，只是平行增加模块。

**范围**：
- `md/codeblock.go`：代码块 segment
  - 检测 fenced code block（` ``` ` / `~~~`）
  - **不调 inline**（代码块里 `*` 不是斜体）
  - 语法高亮直接用 Micro 已有的
  - 可选：背景色区分
- `md/heading.go`：标题 segment
  - 去除 `#` 标记
  - 按级别调整颜色/样式
  - 调 inline 渲染器
- `md/list.go`：列表 segment
  - 美化有序/无序列表缩进和符号
  - 嵌套层级处理
  - 调 inline 渲染器
- 链接 `[text](url)` 加到 inline 渲染器：
  - 只显示 `text`
  - 用特殊颜色标识
- 各自独立配置项 `mdcodeblock` / `mdheading` / `mdlist` / `mdlink`

**Demo**：一份典型 README.md，大部分内容以渲染效果呈现。

**验收标准**：每个 renderer 可独立关闭，关闭后该类内容回退到 paragraph stub。

## v0.5 — mapper（输出行数 ≠ buffer 行数）+ 鼠标精确定位

> **全项目第二复杂的子系统**（仅次于已被探照灯模型消灭的"模式状态机"）。

**目标**：解锁 segment "长得不一样"的能力。表格可以加边框行、代码块可以加标题栏。

**范围**：
- `md/mapper.go`：buffer line ↔ screen line 双向映射
- 表格升级：输出多行（含上下边框行、表头分隔行）
- 代码块升级：加包围框、语言标识栏
- 列表升级：嵌套缩进按视觉层级输出
- 渲染模式下方向键滚动适配新行数差（光标在 segment 内时仍按 buffer 行数移动）
- 鼠标点击定位升级：
  - segment 输出的每个字符分两类：**内容字符**（对应 buffer 真实位置）vs **装饰字符**（边框/padding）
  - 点击内容字符 → 光标定位到对应 buffer 位置
  - 点击装饰字符 → 视为无效点击（不动作）

**Demo**：表格带边框、代码块带框；点击单元格内的文字光标准确定位，点击边框线无效。

**验收标准**：
- mapper 在大文件下性能可接受（目标 < 16ms 每帧）
- 滚动时光标位置追踪正确
- 鼠标点击精度：内容字符 100% 准确

## v0.6 — 10 秒计时器 + 可选"纯阅读模式"（PRD 3.1）

> 风险重新引入版本。但此时架构、所有 renderer、mapper 已全部稳定，可以专心处理输入层。

**目标**：把 PRD 第三章描述的"纯阅读模式"作为**可选增强**加上去。默认仍为探照灯模型。

**范围**：
- 10 秒空闲计时器（配置项 `mdrenderidle`，默认 0 = 关闭 = 永远探照灯模型）
- 当 `mdrenderidle > 0`：
  - 用户 10 秒无操作（无按键、无鼠标点击；鼠标移动不算）→ 进入纯阅读模式
  - 纯阅读模式：光标隐藏，方向键 = 滚动，按任意键 → 退出（回到探照灯模型 + 该键正常处理）
- 输入事件拦截层（这是全项目风险最高的代码）
- 渲染模式下方向键滚动（基于 mapper 之后的屏幕行数）
- per-pane 独立计时器

**Demo**：配置 `mdrenderidle: 10`，闲置 10 秒后光标消失，方向键变成滚动。按键立刻回到探照灯。

**验收标准**：
- `mdrenderidle: 0` 时行为与 v0.5 完全一致
- `mdrenderidle > 0` 时所有 Micro 快捷键在阅读模式下仍可工作（`Ctrl+S`、`Ctrl+Q`、`:e` 等）
- 切换 pane 时各 pane 计时器独立

**待拍板**：拖选复制的具体内容形式（渲染后纯文本 vs 原始 markdown，装饰字符如 `|` 如何处理）。

## 附：与 PRD 第三章的关系

PRD 第三章描述的"双模式模型"在本方案下被重新定位：

| PRD 概念 | 本方案落地 |
|---------|----------|
| 渲染模式（默认） | 探照灯模型（光标所在行回退原生） |

- 还是把双模式描述整体替换为探照灯模型
- 或者两者并存，由用户选择

## 待讨论

1. v0.1 的范围是否过大？目前看是项目里**最重的一版**，因为要同时落地架构 + 探照灯 + 100% 接管 + 行号移除。如果过大，可拆为：
   - v0.1a：架构 + 100% 接管（segment 全是 stub，光标行为不变）
   - v0.1b：探照灯（光标驱动 segment fallback）+ 行号移除
2. `paragraph` segment 的 Render() 何时被实际调用？理论上光标在它范围内时回退原生，所以用户永远看不到 paragraph 的输出。但 segment 不可见时（光标在其他 segment，paragraph 在可视范围）会被调用——这是它的实际工作场景。
3. v0.6 引入"纯阅读模式"后，探照灯还是默认吗？建议是。纯阅读模式只对 `mdrenderidle > 0` 的用户生效，默认行为始终是探照灯。
