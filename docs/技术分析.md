## 一、Micro 渲染架构

### 核心流程

```
用户按键 → action 层处理 → 修改 buffer → 主循环触发 → display 层渲染 → screen 输出
```

具体来说（`cmd/micro/micro.go` 主循环）：

```
1. screen.Fill(' ')        // 清屏
2. Tabs.Display()          // 标签栏
3. ep.Display()            // 每个编辑面板 ← 我们的主战场
4. MainTab().Display()     // 标签窗口
5. InfoBar.Display()       // 底部信息栏
6. screen.Show()           // 刷新终端
```

### 关键模块

| 模块 | 职责 | 我们要改？ |
|------|------|-----------|
| `internal/display/bufwindow.go` (903行) | 缓冲区内容渲染 | **✅ 主要改这里** |

## 二、渲染流程详解（displayBuffer 函数）

`bufwindow.go` 的 `displayBuffer()` 是核心渲染函数，903 行中占了约 400 行：

```
1. 计算窗口参数（gutter宽度、行号宽度等）
2. 逐行遍历 buffer：
   a. 画 gutter（消息、diff标记、行号）
   b. 画正文：
      - getStartInfo() 定位到水平滚动后的起始位置
      - 逐字符解码，计算视觉宽度（tab展开、宽字符）
      - getStyle() 获取语法高亮样式
      - screen.SetContent(x, y, rune, style) 逐格画到屏幕
   c. 处理软换行（softwrap/wordwrap）
3. 填充行尾空白
4. 处理光标显示
```

### 关键数据结构

```go
// 屏幕上的视觉位置
vloc = buffer.Loc{X, Y}    // X: 窗口内像素列, Y: 窗口内行号

// buffer 中的字符位置
bloc = buffer.Loc{X, Y}    // X: 行内第N个字符, Y: 第N行

// 软换行中的位置
SLoc = {Line, Row}         // buffer 行号 + 该行内的视觉行序号
VLoc = {SLoc, VisualX}     // 加上水平视觉坐标
```

### 关键 API

```go
// 把一个字符画到屏幕上（底层）
screen.SetContent(screenX, screenY, rune, combc, style)

// buffer → 视觉位置
w.VLocFromLoc(bufferLoc) → VLoc

// 视觉位置 → buffer 位置（鼠标点击时用）
w.LocFromVisual(visualLoc) → buffer.Loc
w.LocFromVLoc(VLoc) → buffer.Loc
```

## 三、表格对齐的实现方案

### 核心思路

**只改显示，不改数据。** Buffer 里存的是原始 Markdown（不对齐的），渲染时临时计算对齐后的显示。

### 需要改动的 3 个层次

#### 层次 1：表格检测与列宽计算（新模块）

新建 `internal/display/markdown_table.go`：

```go
// TableBlock 表示一个连续的表格区域
type TableBlock struct {
    StartLine int       // 表格起始行
    EndLine   int       // 表格结束行
    ColWidths []int     // 每列的最大宽度
}

// 检测并解析 buffer 中的表格块
func DetectTables(buf *buffer.Buffer, startLine, endLine int) []TableBlock
```

**检测规则**：
- 连续 ≥ 2 行，每行以 `|` 开头且以 `|` 结尾
- 或 `|` 分隔符 ≥ 2 个
- 分隔行（`|---|---|`）用来确认是表格而非普通竖线文本

**列宽计算**：
- 遍历表格所有行，按 `|` 分割
- 每列取最大宽度
- 分隔行宽度跟随内容

#### 层次 2：渲染替换（改 displayBuffer）

在 `displayBuffer()` 的逐行渲染循环中，加入表格检测分支：

```
原始流程：
  for 每行 → 逐字符 → screen.SetContent

改后流程：
  for 每行 {
      if 这行属于某个 TableBlock {
          用对齐后的内容渲染（补空格，列对齐）
      } else {
          原始渲染逻辑
      }
  }
```

**关键**：渲染时对每行"虚拟地"在 `|` 之间补空格，使列宽一致。**不修改 buffer 中的实际数据**。

#### 层次 3：光标坐标映射（改 softwrap.go 的相关函数）

这是**最关键也最复杂**的部分。

问题：屏幕上显示的是对齐后的内容（多了空格），但 buffer 里存的是原始内容。光标移动时需要双向转换。

```
屏幕坐标 → buffer 坐标（点击/移动光标时）
  屏幕X=15 → 表格第2列偏移3 → buffer中第2列偏移1（原始内容更短）

buffer 坐标 → 屏幕坐标（渲染光标时）
  buffer X=10 → 表格第2列偏移1 → 屏幕X=18（对齐后位置更靠右）
```

需要修改的函数：
- `VLocFromLoc()` / `getVLocFromLoc()` — buffer位置 → 屏幕位置（加偏移）
- `LocFromVLoc()` / `getLocFromVLoc()` — 屏幕位置 → buffer位置（减偏移）
- `getStartInfo()` — 水平滚动起点计算

## 四、架构设计原则

### 1. 最小侵入

```
新增文件：
  internal/display/markdown_table.go  (~200行)  表格检测+列宽计算

修改文件：
  internal/display/bufwindow.go       (~200行)  渲染分支+光标映射
  internal/display/softwrap.go        (~100行)  坐标转换适配

不动：
  buffer/ 层全不动
  screen/ 层全不动
  action/ 层全不动
```

### 2. 可开关

添加配置项 `mdtablealign`（默认开启），方便用户关闭回归原始行为。

### 3. 性能考虑

- 表格检测结果应**缓存**，仅在该行区域被修改时重新计算
- `ModifiedThisFrame` 标志已经存在，可以复用
- 只检测可视区域内的行（`StartLine` 到 `StartLine + bufHeight`）

### 4. 渐进实现顺序

```
Phase 1: 只读对齐（不做光标映射）
  - 检测表格 + 对齐渲染
  - 光标映射先不管，编辑时临时关闭对齐
  - 验证显示效果

Phase 2: 完整光标映射
  - 双向坐标转换
  - 编辑时实时重新对齐

Phase 3: 优化
  - 缓存表格检测结果
  - 边界情况处理
```

## 五、风险点

| 风险 | 影响 | 应对 |
|------|------|------|
| 坐标映射错误 → 光标乱跳 | 高 | Phase 1 先跳过，Phase 2 大量测试 |
| 表格检测误判（非表格被当成表格） | 中 | 用分隔行（`\|---\|`）确认 |
| 性能问题（大文件每帧检测表格） | 低 | 只检测可视区域 + 缓存 |
| 软换行与表格对齐交互 | 中 | 表格行可以考虑自动禁用软换行 |
