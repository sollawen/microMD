## 问题 1：displayBuffer 和 Pipeline 的边界（最关键）

### 它是什么

Micro 里有一个核心函数叫 `displayBuffer()`，位于 `internal/display/bufwindow.go`，**946 行长**。它是 Micro 的"主渲染函数"——每帧屏幕刷新都会调一次。

它的结构大致是：

```go
func displayBuffer() {
    // 1. 画顶部状态栏、标签栏
    // ...

    // 2. 主循环：逐行画 buffer 内容
    for 每个可见行 line {
        // 2a. 画 gutter（行号、消息标记）
        // 2b. 画内容：
        //     逐字符读取 buffer 这一行
        //     处理 tab、宽字符、软换行、syntax highlighting
        //     调用 screen.SetContent 把字符写到屏幕缓冲
        // 2c. 画光标
    }

    // 3. 收尾
}
```

我们的目标是：让"光标在 segment 里时这一行用 Micro 原生渲染，光标不在时这一行用我们的 segment 渲染"。两种渲染需要**共存于同一个屏幕**。

问题是：**我们的代码插入到 `displayBuffer()` 的哪个位置？**

### 两种选项

#### 方式 1：增量式（在 Micro 的循环里加分支）

保留 `displayBuffer()` 的整体结构不变，在主循环里加一个 if：

```go
for 每个可见行 line {
    if 是.md文件 && mdrender开启 && !segment.IsEditState(cursors) {
        // 走我们的 Pipeline
        cells := pipeline.RenderLine(line)
        for cell in cells {
            screen.SetContent(...)
        }
    } else {
        // 走 Micro 原来的 946 行逻辑
    }
}
```

**特点**：
- 改动小（~50 行）
- 外层循环还是 Micro 的
- 我们只是"在某些行的内容生成上插一脚"
- gutter、光标、视口控制都还是 Micro 处理

#### 方式 2：替换式（我们的 Pipeline 接管整个循环）

```go
func displayBuffer() {
    if 是.md文件 && mdrender开启 {
        // 全权委托给 Pipeline
        pipeline.RenderEntireDisplay(...)
        return
    }
    // 否则走 Micro 原本的逻辑
}
```

**特点**：
- 改动大（~300 行，要重写"for 每行"循环）
- 我们的代码是独立的、完整的、可读的
- Pipeline 内部对每行处理更灵活

### Trade-off

| 维度 | 方式 1（增量式） | 方式 2（替换式） |
|------|---------------|---------------|
| v0.1 编码量 | 小（~50 行） | 大（~300 行） |
| 架构清晰度 | 低（Micro 和我们代码缠在一起） | 高（边界明确） |
| gutter / 光标由谁画 | 自然由 Micro 画 | 我们要自己画（或调 Micro 子函数） |
| Micro 上游合并 | 容易（冲突局限在 if 分支） | 困难（我们重写了循环） |
| 未来加渲染特性 | 受限（要在 Micro 循环里塞） | 自由（自己的循环） |
| 调试 | 难（要看 Micro 整个 displayBuffer） | 容易（看 Pipeline 就行） |

### 我的建议

**方式 1（增量式）**。理由：

1. Micro 的 `displayBuffer()` 太复杂，946 行。重写它（方式 2）风险大。
2. 方式 1 让 gutter、光标、视口、softwrap 都自然由 Micro 处理——这就自动解决了问题 2。
3. Micro 上游合并成本低。Micro 不是死的，将来可能更新。
4. v0.1 阶段不需要"灵活处理每行"的高级能力，方式 1 够用。

代价是架构清晰度低、未来加特性时受限。但等到 v0.5/v0.6 真需要时，再从方式 1 演进到方式 2 也来得及。

## 问题 2：`.md` 文件的 gutter 和 softwrap 由谁负责

> 这个问题很大程度上被问题 1 的选择决定。如果选方式 1，自动解决大半。

### 什么是 gutter

Gutter 是编辑器左侧那一列，包含：
- **行号**（`1`、`2`、`3`...）
- 消息标记（lint 错误、git 修改标记）
- 当前行的特殊高亮背景

PRD 5.5 已明确：**`.md` 文件不显示行号**。

但 gutter 不只是行号。即使去掉了行号，Micro 的 `displayBuffer` 还可能画：
- diff gutter（git 修改标记，由 `diffgutter` 配置控制）
- message gutter（语法错误等）

这些要不要在 `.md` 文件里显示？PRD 没明说。

### 什么是 softwrap

Softwrap = 软换行。当一个 buffer 行的字数超过屏幕宽度时：

- **不开启**：行向右延展，用户用 ←→ 滚动看完整内容
- **开启**：行在屏幕边缘自动折到下一屏幕行（buffer 还是 1 行，但屏幕占多行）

PRD 5.6 明确：v0.1–v0.5 阶段，**渲染片内不换行，超宽内容向右延展**。

### 为什么是问题

**Gutter 问题**：如果我们的 Pipeline 也画 gutter，会和 Micro 原生 fallback 的 gutter 冲突（两边都画）；如果 Pipeline 不画 gutter，那走 Pipeline 路径时 gutter 就缺失。

**Softwrap 问题（更严重）**：

考虑这个场景——一个表格的标题行很长：

```
光标不在表格 → segment 渲染 → 不软换行 → 屏幕占 1 行
光标进入表格 → Micro fallback → Micro 检查 softwrap 配置
              ├─ softwrap 开启 → 屏幕占 3 行
              └─ softwrap 关闭 → 屏幕占 1 行
```

如果用户全局开了 softwrap，**同一行从 segment 状态切到 fallback 状态时，屏幕高度会从 1 行变 3 行**——画面跳变，Mapper 记录的行数也错乱。

### 解决方案

#### 选项 A：对 `.md` 文件强制禁用 softwrap（和 gutter）

在 Host Integration 层判断文件类型，对 `.md` 文件：
- `softwrap = false`（无视用户全局配置）
- `ruler = false`（行号关掉，PRD 已定）
- `diffgutter` 和 `msg gutter`：待定（建议保留，diff 信息有用）

**优点**：简单粗暴，符合 PRD 5.6 早期策略。
**缺点**：用户在 `.md` 文件里配置的 `softwrap` 失效。但这是合理的——用户选了 MicroNeo 就是来看渲染效果的，软换行和渲染本身就有冲突。

#### 选项 B：让 segment 输出也支持 softwrap

让我们的 segment 渲染时也按窗口宽度软换行，保持和 Micro fallback 一致。

**缺点**：复杂，要和 Micro 的 softwrap 算法保持一致。v0.1 阶段不推荐。

### 我的建议

**选项 A**。`.md` 文件强制：
- `softwrap = false`
- `ruler = false`
- `diffgutter` 和 `msg gutter` 保留（待 v0.1 实施时再讨论）

PRD 5.6 明确"渲染片内不换行"是早期策略，所以禁用 softwrap 是符合产品规划的。

## 问题 4：建立 v0.1 性能基准

### 它是什么

架构文档说"每帧重渲染，性能不是问题"，但没有数据证明。

Lisa 担心：v0.1 阶段 Pipeline 每帧都要做：

1. 跑 Detector 识别 segment（视口范围内）
2. 对每个可见行调 Segment.Render()
3. 把输出写到 screen

如果任何一步意外变慢，用户感知就是"打字延迟""滚动卡顿"。这种 bug 很难定位（是 Detector 慢？是 Render 慢？是 screen 写入慢？）。

### Lisa 的建议

在 v0.1 阶段就：

1. 写一个 benchmark：视口 80 行、文件 10K 行，测 `Pipeline.Render()` 的耗时
2. 把这个 benchmark 加进 CI
3. 任何改动导致耗时增加 > 10%，CI 失败

### 我的看法

这条不算紧急。v0.1 阶段所有 segment 都是 stub，没有真渲染逻辑，性能肯定没问题。真正需要担心是 v0.2 表格真渲染之后。

但 Lisa 的建议**成本低、收益高**：写一个 benchmark 文件大概 10 分钟，加到 CI 后永久受益。

### 我的建议

- 不作为 v0.1 的硬性验收标准（不阻塞 v0.1 发版）
- 但作为 v0.1 阶段"应该做"的事
- v0.2 加表格真渲染时，benchmark 会真正发挥作用
