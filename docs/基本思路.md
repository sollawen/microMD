# 我的总体想法

我觉得我们的microMD，可以走一条 渲染与实际的editing buffer分开的路。
- display的时候，把markdown文件渲染得更漂亮。但是不涉及原始的md文件里的原始的每行的文字。仅是渲染
- 当用户光标移动到某行的时候，说明用户想编辑这行的文字了。
	- 这时，这行或上下关联的若干行，就脱离渲染状态，只显示micro原来的显示和编辑方式
- 当用户光标移走后，这行或上下关联的若干上行，就又进入渲染的显示模式

这样做的好处是：
- 渲染就是渲染，渲染的时候，不关心光标的坐标转换
- 一旦光标移进来了，就脱离渲染的显示模式，回归到micro原生的显示和编辑模式里，代码不用改

## 举例

比如表格：
- 表格一定是由相邻的若干行组成的
- 因此，平时在渲染模式里，当程序发现是表格了，就专门有代码做表格的渲染
	- 我们已经实现了一个最基本的表格渲染功能
	- 以后甚至可以用ascii的表格线段字符，把这个表格渲染的更漂亮些

当用户把光标移入这几行的时候，就脱离渲染模式了
- 就回到了micro最原生的显示和编辑模式
- 用户的输入和删改，我们都不去渲染，编辑模式和渲染没有关系
- 直到用户的光标脱离了这个表格的区域后，才恢复到渲染模式

比如粗体字和斜体字
- micro原生的显示，只是用不同的颜色来区别了，并没有真的显示成粗体和斜体
- 没关系。只要用户的光标不在这行上，这行就是渲染模式。
	- 这时就不显示什么星号了。而是真的显示粗体和斜体，
	- 当然颜色还是使用micro自己theme的对应颜色
- 当用户光村移到这行的时候，马上回到micro的原生显示和处理逻辑里，不渲染了

## 未来的逐渐扩展

比较我们一开始，只有一个表格的渲染模块
- 然后，我们又加上了一个单行的粗体、斜体的渲染模块
- 然后，我们又加上一个代码块的渲染模块（处理应该是和表格的逻辑类似的）
- 然后，我们又加上了......


# 几个问题

## 1，渲染行数与原始行数的对齐

我的想法是：

1，渲染片根本就不要对齐
	- 什么是渲染片
		- 比如表格模块，它只负责这个表格的渲染。那么这个表格就是一个渲染片。
		- 比如代码块，它只负责这个代码块的渲染。那么这个代码块就是一个渲染片
	- 对一个渲染片，一定是原始md里面的相邻的若干行
		- 所以，一定有一个 lineStart and lineEnd,在这两个行号之间的内容就是这个渲染片
		- 比如 line 5-10，是个表格，我为了漂亮，我用了8行来渲染它。那么这8行的渲染片就对应着 line5-10
	- 当光标进入 line5-10的时候，渲染片就消失了，变成了原来的micro的逻辑

比如文件有20行，其中 line 5-10 是表格，渲染成了 8 行。那么屏幕上实际显示的不是
 20 行，而是 23 行（20 - 6 + 8 = 22）。

 这时，屏幕的显示就应该是这样的

 最左边是行号


2，单行的渲染，肯定是和原始的单行的行号，一一对应的
	- 渲染片中间的每行也是有单行的渲染的
