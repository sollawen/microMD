## 1. 思考出发点

"项目要做什么"翻译成架构需要回答的几个根本问题：

1. **每一帧怎么决定屏幕上每个 cell 显示什么？**（数据流）
2. **新增一种 Markdown 元素的渲染（比如代码块）需要改几个地方？**（扩展性）
3. **关闭某个渲染器（`mdcodeblock: false`）后行为如何退化？**（配置可组合性）
4. **`.md` 文件和非 `.md` 文件的代码路径如何分流？**（边界隔离）
5. **鼠标点击怎么从屏幕坐标回到 buffer 位置？**（反向数据流）

下面这套架构就是冲着回答这 5 个问题设计的。

## 2. 渲染模型：每帧重渲染

> 这是架构最基础的选择。所有后续设计都基于此。

### 2.1 模型定义

采用**每帧重渲染**模型：

```
没有"已渲染好的整篇文档"这种持久化数据。

每帧（屏幕刷新）：
  1. 看视口当前指向哪些 buffer 行（StartLine）
  2. Pipeline.Render(这些行) → 当帧的屏幕内容
  3. 写到 screen
  4. 帧结束，输出丢弃

scroll 发生时：
  - Micro 把 StartLine +1
  - 下一帧重新走 1-2-3
  - 视觉上看起来像"内容往上滚"，但其实是"重新画了一遍"
```

### 2.2 为什么是这个模型

1. **Micro 自己就是这种模型**。我们的代码插入它的渲染循环里，沿用同一种模型最自然。
2. **编辑体验更直接**。用户改一行 → 下一帧自动反映正确，没有缓存失效范围的 bug 风险。
3. **大文件友好**。10K 行的 .md，每帧只渲染视口里那 50 行，不需要全量预渲染。
4. **性能足够**。视口通常 40-80 行，每行渲染 + 写 screen 在 1ms 量级，远低于 16ms 帧预算。

### 2.3 输出目的地：screen 层，不是 pane buffer

Micro 实际的渲染链路：

```
Micro 主循环
  → Tabs.Display()
    → 每个 pane 的 Display()
      → displayBuffer()             ← internal/display/bufwindow.go
        → for 每行：
            for 每字符：
              screen.SetContent(x, y, rune, style)   ← internal/screen/screen.go
  → screen.Show()                   ← 把 tcell 后台缓冲刷到终端
```

**Micro 里没有 "pane buffer" 这个概念**。每个 pane 的渲染是直接把字符 cell 写到 screen 层（实质是 tcell 的后台缓冲）。主循环最后调一次 `screen.Show()` 统一刷出。

我们的 Pipeline 输出（`[]RenderCell`）也写到同一个地方：

```
displayBuffer()（改造后）
  ├─ 非 .md 文件 → 走 Micro 原来的逐 rune 循环 → screen.SetContent
  └─ .md 文件：
       Pipeline.Render() → []RowOutput
       for 每行的 RenderCell：
         screen.SetContent(x, y, cell.Rune, cell.Style)
```

"Micro 原生渲染"和"我们的 segment 渲染"只是**两条产生 `(rune, style)` 序列的不同代码路径**，它们写到同一个目的地（screen）。所以 edit state 的 fallback 没有"切换显示后端"的复杂度——只是改用另一段代码生成 cell 而已。

Pipeline 输出 `[]RenderCell` 看起来像是个中间层，但它**只是函数返回值**，生命周期就在一帧之内——生成完立刻被消费掉、写到 screen，不持久化。

## 3. 分层架构（5 层 + 横切）

```
┌─────────────────────────────────────────────────────┐
│  Micro 主循环（不动）                                │
│  ┌───────────────┐  ┌───────────────┐              │
│  │ action 层     │  │ buffer 层     │              │
│  │ (按键 → 编辑) │  │ (文件数据)    │              │
│  └───────────────┘  └───────────────┘              │
└─────────────────────┬───────────────────────────────┘
                      │ 接入点：bufwindow.go displayBuffer()
                      ▼
┌─────────────────────────────────────────────────────┐
│  Layer 5: Host Integration（宿主集成层）              │
│  - 判断是否 .md / 配置是否开                          │
│  - 否 → 完全交给 Micro 原生显示，我们的代码不执行     │
│  - 是 → 调用 Pipeline，并把最终输出通过               │
│         screen.SetContent 写到 screen 层             │
└─────────────────────┬───────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────┐
│  Layer 4: Pipeline（渲染管线 / 总指挥）               │
│  - 输入：buffer、可视范围、光标列表、配置              │
│  - 输出：屏幕每行的 RenderCell[]                      │
│  - 职责：协调下面三层，决定每行由谁处理               │
└──┬──────────┬──────────┬──────────┬─────────────────┘
   ▼          ▼          ▼          ▼
┌─────────────────────────────────────────────────────┐
│  Layer 3: Segment Registry（渲染片注册中心）           │
│  - 知道当前 buffer 里有哪些 segment、各自的范围        │
│  - 对外提供：given line → which segment?              │
│  - 内部维护：检测、缓存、失效                          │
└─────────────────────┬───────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────┐
│  Layer 2: Segment Renderers（各结构渲染器）           │
│  table / codeblock / heading / list / blockquote     │
│  / hr / paragraph（兜底）                             │
│  - 每个 renderer 回答：                                │
│    1) 我能处理这块吗？（detect）                       │
│    2) 输出是什么？（render）                           │
│  - 调用 Layer 1 处理单元格/标题内的行内 markdown       │
└─────────────────────┬───────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────┐
│  Layer 1: Inline Renderer（行内渲染器 / 公共工具）    │
│  - 输入：一行 markdown 文本                            │
│  - 输出：去标记后的文本 + 每字符的终端属性              │
│  - 处理：粗体、斜体、行内代码、链接                     │
│  - 例外：代码块 segment 不调用它                       │
└─────────────────────────────────────────────────────┘
```

### 横切关注点（不属于某一层，被各层共享）

```
┌─────────────────────────────────────────────────────┐
│  Mapper（坐标映射层）                                 │
│  - buffer line ↔ screen line 双向映射                 │
│  - 解决"buffer 与 screen 之间的行数对应关系"           │
│  - v0.1–v0.4 阶段是 1:1（passthrough），代码也存在    │
│  - v0.5 起 segment 输出多行才真正起作用                │
│  - 详见第 5 章                                       │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  Config（配置层）                                     │
│  - 所有渲染器在启动时读取一次自己的配置                 │
│  - 配置变更时触发 segment registry 失效缓存            │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  Syntax Highlight Adapter（语法高亮适配）              │
│  - 包装 Micro 已有的高亮系统                           │
│  - 给定 buffer 位置，返回颜色                          │
│  - Layer 1 在输出时叠加颜色到终端属性                  │
└─────────────────────────────────────────────────────┘
```

## 4. 一帧的数据流

```
Micro 主循环触发 displayBuffer()
        │
        ▼
Layer 5 (Host Integration):
  这是 .md 文件？mdrender 开关开了？
        ├─ 否 → Micro 原生路径，return
        └─ 是 → 继续
        ▼
Layer 5: 调用 Pipeline.Render(buf, viewport, cursors, config)
        ▼
Layer 4 (Pipeline):
  1. 用 Mapper 把 viewport 起点（buffer 坐标）
     转成"应该从哪些 buffer 行开始"
  2. for 每个相关 buffer 行 line:
       a. segment = Registry.GetSegmentAt(line)
       b. editState = segment.IsEditState(cursors)
       c. if editState:
            output = Micro 原生逐 rune 渲染该行
          else:
            output = segment.Render(line)
       d. 把 output 写到 screen（逐 cell 调用 screen.SetContent）
       e. 告诉 Mapper 这行实际占用了多少 screen 行
  3. Pipeline 返回
        ▼
Layer 5: 控制权交回 Micro 主循环
Micro 主循环: screen.Show() 把缓冲刷到终端
```

### 关键设计点

- **Pipeline 对每一行问 Registry 同一个问题**（"这行谁负责"），它不关心 segment 内部结构。
- **edit state 决策权在 segment**，不在 Pipeline。Pipeline 只问 segment：你现在是编辑态吗？未来加多光标、加 segment 子区域回退等逻辑，都在 segment 内部扩展。
- **fallback 到 Micro 原生** 是 Pipeline 的一个分支，不是异常路径。Pipeline 显式选择这条路。
- **两条路径最终都汇合到 `screen.SetContent`**——Micro 原生渲染和我们的 segment 渲染只是产生 `(rune, style)` 序列的不同代码，目的地相同。
- **每帧重新计算**——Pipeline 的输出（`[]RenderCell`）是临时数据，写完 screen 就丢弃，不持久化（见第 2 章渲染模型）。

## 5. Mapper：buffer 与 screen 的行对应关系

> Mapper 的存在是从基本需求推导出来的，不是凭空加的层。本章解释它的来由、职责、与 Segment 的分工。

### 5.1 为什么需要 Mapper

原始 markdown 文件（buffer）由若干行组成，screen 是一个 x×y 矩阵。renders 的作用是把 buffer 某行的内容**计算后**填到 screen 里去。

但 buffer 行与 screen 行不是一一对应：

```
buffer 第 N 行  →  screen 上占几行？
                    ├─ 1 行（短段落，无 softwrap）
                    ├─ 多行（长段落 + softwrap）
                    ├─ 0 行（被隐藏，比如折叠）
                    └─ 多行（表格里某行因为单元格内容换行了）
```

而且，scroll 发生时，Micro 给我们的是"屏幕起始位置"，我们要反推：

```
屏幕第 0 行  ←  是 buffer 的哪一行？是某个 segment 输出的第几行？
```

考虑这个场景：用户按 ↓ 滚动一行，我们怎么知道下一帧从哪个 buffer 行开始渲染？

```
当前状态：viewport 从 (buffer 第 100 行 + segment 输出的第 0 行) 开始
按 ↓：
  ├─ 如果第 100 行的 segment 输出还有更多行 → viewport 内部偏移 +1
  └─ 如果第 100 行的 segment 输出已经到底 → 跳到 buffer 第 101 行的 segment 输出第 0 行
```

**这就需要"buffer 第 100 行的 segment 占多少 screen 行"的信息。**

如果不维护这个信息，每次滚动都要从 buffer 第 0 行开始算累加——10K 行的文件就完了。

### 5.2 Mapper 的双向职责

**正向映射**（buffer → screen）：

```
buffer 第 N 行 + 该行 segment 输出的第 K 个屏幕行  →  screen 的第 M 行
```

用于：渲染光标位置、把 buffer 位置标在高亮、tooltip 定位等。

**反向映射**（screen → buffer）：

```
screen 第 M 行  →  buffer 第 N 行 + 该行 segment 输出的第 K 个屏幕行
```

用于：
- 滚动时计算下帧 viewport 起点
- 鼠标点击定位（先把屏幕 click 点转成 buffer 位置，再决定光标落在哪里）

### 5.3 Mapper 与 Segment 在处理跨行结构上的分工

这是容易混淆的一点。Markdown 里有三类跨行结构（代码块、表格、引用块），它们的"上下文相关"由谁处理？

**答案：由 Segment 处理，不是 Mapper。**

Segment 是"需要上下文的一组行的自然单元"。Registry 把 buffer 切分成 Segment 序列：

```
buffer 第 100-104 行 → TableSegment{StartLine: 100, EndLine: 104}

TableSegment.Render(100) ← 知道自己是表头
TableSegment.Render(101) ← 知道自己是分隔行
TableSegment.Render(102) ← 知道自己是数据行
```

Pipeline 在问"第 100 行谁负责"时，TableSegment 跳出来，它内部已经持有 5 行的完整信息，所以单行 Render 也能正确反映上下文。

Mapper 不关心内容含义，只关心"这行渲染完后占了几个屏幕行"。

**分工总结**：

| 关注点 | 由谁处理 |
|-------|---------|
| 这一行是表格的第几行？应该怎么画？ | Segment（跨行结构） |
| 这一行的内容跨行了吗？跨成几行？ | Segment 输出 + Mapper 记录 |
| 视口应该从哪个 buffer 行开始？ | Mapper（反向） |
| 鼠标点击屏幕第 M 行，对应 buffer 哪里？ | Mapper（反向）+ Segment（精确定位到字符） |

### 5.4 Mapper 的生命周期：每帧重算，但状态帧间持久

这是和第 2 章"每帧重渲染"模型一致的：

```
每帧流程：
  1. Pipeline 开始
  2. 用 Mapper 算出视口涉及的 buffer 行范围（基于上帧的 viewport 状态）
  3. 对这些行调 Segment.Render()
  4. 顺便记录每行实际输出多少 screen 行（更新 Mapper 本帧的状态）
  5. 写到 screen
  6. 帧结束，Mapper 的状态保留到下帧（用于下帧的起点计算）
```

Mapper 的状态在帧之间持久，但内容（每行多少 screen 行）会随 buffer 变更失效。这和 Segment 一样，是缓存式的状态。

### 5.5 Mapper 在 v0.1–v0.4 是 passthrough

重要：

- **v0.1–v0.4**：所有 segment 输出 1:1（一 buffer 行 = 一 screen 行）。Mapper 的实现就是 `screenLine == bufferLine`，几行代码。
- **v0.5+**：表格加边框、代码块加框、段落 softwrap 等，才让 Mapper 真正工作。

**Mapper 这个模块从 v0.1 就存在，只是早期它就是个恒等函数**。这样架构是连续的，不是 v0.5 突然引入新东西。

### 5.6 Mapper 的接口（草案）

```go
type Mapper struct {
    // 缓存：buffer 行号 → 该行占用的 screen 行数
    lineHeights map[int]int
    // ...
}

// 正向：buffer 位置 → screen 行号
func (m *Mapper) BufferToScreen(line, rowInSegment int) int

// 反向：screen 行号 → buffer 位置
func (m *Mapper) ScreenToBuffer(screenLine int) (bufferLine, rowInSegment int)

// 帧内更新：渲染完一行后告诉 Mapper 这行占多少 screen 行
func (m *Mapper) RecordLineHeight(bufferLine, height int)

// 失效：buffer 变更时
func (m *Mapper) Invalidate(startLine, endLine int)
```

## 6. 关键模块的职责与接口契约

### Layer 5: Host Integration（宿主集成层）

**职责**：判断要不要启动 MicroNeo 渲染，并把控制权转交（或不转交）给 Pipeline。

**位置**：`internal/display/bufwindow.go`，入口在 `displayBuffer` 函数最前面。

**接口**：只是一个判断 + 委托，不暴露新 API。

**输出目的地**：所有渲染结果（无论来自 Pipeline 还是 Micro 原生 fallback）都通过 `screen.SetContent(x, y, rune, style)` 写到 screen 层。**没有 pane buffer 这种中间层。**

### Layer 4: Pipeline（渲染管线）

**职责**：每帧主控。

**接口**（草案）：

```go
type Pipeline struct {
    registry *Registry
    mapper   *Mapper
    // ...
}

func (p *Pipeline) Render(
    buf *buffer.Buffer,
    viewport Viewport,
    cursors []*buffer.Cursor,
) []RowOutput
```

`RowOutput` 是一行屏幕内容的完整描述（runes + styles + 内容/装饰标记）。

### Layer 3: Segment Registry（渲染片注册中心）

**职责**：知道当前 buffer 里所有 segment 的分布。

**接口**（草案）：

```go
type Registry struct { ... }

func (r *Registry) GetSegmentAt(line int) Segment
func (r *Registry) Invalidate(startLine, endLine int)  // buffer 变更时
func (r *Registry) Rebuild(buf *buffer.Buffer)         // 全量重建
```

**实现要点**：内部维护一个按 `lineStart` 排序的 segment 列表，二分查找。对"光标所在的 segment"用同样的查找。

**注册流程**：Registry 持有一组 `Detector` 函数，每个 Detector 负责识别一类 segment：

```go
type Detector func(buf *buffer.Buffer, startLine, endLine int) []Segment
```

例如 `TableDetector`、`CodeblockDetector`、`ParagraphDetector`（兜底）。

启动时按顺序跑所有 Detector，先识别 specialty segment（表格、代码块等），剩下的行由 paragraph 兜底。

### Layer 2: Segment（渲染片）

**职责**：负责一段连续 buffer 行的渲染。

**接口**（草案）：

```go
type Segment interface {
    StartLine() int
    EndLine() int
    Contains(line int) bool

    IsEditState(cursors []*buffer.Cursor) bool

    // v0.1–v0.4：输出 = buffer 行数（1:1）
    // v0.5+：可能输出多行（含装饰行）
    Render(line int) []RenderCell
}

type RenderCell struct {
    Rune  rune
    Style tcell.Style
    Kind  CellKind  // Content / Decoration / Padding
}
```

**CellKind** 是为 v0.5 鼠标点击准备的字段——v0.1 阶段可以全填 `Content`，不影响行为。

### Layer 1: Inline Renderer（行内渲染器）

**职责**：处理行内 markdown。

**接口**（草案）：

```go
type InlineRenderer struct { ... }

func (r *InlineRenderer) Render(
    text []byte,
    baseStyle tcell.Style,
) []RenderCell
```

**关键约束**：跨行闭合保守处理（标记未闭合则 passthrough）。Inline Renderer **不调用** Syntax Highlight Adapter；高亮叠加在更上层（Pipeline 或 Segment 内）做。

## 7. 关键设计决策（待逐条确认）

### D1：Segment 是"被动数据"还是"主动对象"？

**提案**：主动对象。

每个 segment 自己持有 `StartLine` / `EndLine`、自己实现 `IsEditState`、自己实现 `Render`。Registry 只是一个索引层。

**理由**：扩展新 renderer 时，只需要在 `md/` 下加一个新文件实现 Segment 接口 + 注册一个 Detector，**不需要改 Registry 或 Pipeline**。

### D2：paragraph segment 真的存在，还是 Pipeline 的兜底分支？

**提案**：真的存在。

Paragraph 作为一个具体的 Segment 类型实现，对所有未匹配 specialty 的行负责。

**理由**：保证 Pipeline 的代码路径单一——它永远调 `segment.Render()`，没有 "if 没有 segment then fallback" 这个分支。架构对称性更好。

### D3：edit state 下"回退 Micro 原生"是谁的责任？

**提案**：Pipeline 的责任。

Segment 只回答"我是不是 edit state"，不关心"edit state 时谁来渲染"。Pipeline 拿到 `IsEditState: true` → 调用 Micro 原生的逐 rune 渲染路径。

**理由**：Segment 只懂自己的渲染，不应当懂 Micro 原生渲染。职责隔离。

### D4：Mapper 是 Pipeline 内部，还是独立横切层？

**提案**：独立横切层，被 Pipeline 调用。

mapper 的输入是"segment 输出列表"，输出是"屏幕行 ↔ buffer 行的映射"。它是一个纯函数式变换，独立成层便于测试、便于将来替换算法。

v0.1–v0.4 阶段 mapper 是 passthrough（segment 输出几行就是几行），v0.5 起 segment 可以输出多行，mapper 才真正工作。

详见第 5 章。

### D5：配置变更如何传播？

**提案**：通知 + 失效模式。

配置变更 → 通知 Registry `Invalidate()` → 下一帧重建 segment。

不做实时传播（不强制正在渲染的 segment 中途读取新配置），避免渲染中途状态不一致。

### D6：与 Micro 高亮系统的边界

**提案**：通过适配器调用，不直接耦合。

新建 `md/highlight.go` 封装对 Micro 高亮的访问。

**理由**：未来如果 Micro 高亮系统改了，我们只改一处。同时，我们渲染时是否要叠加高亮、要不要过滤某些颜色，都可以在适配器里决策。

## 8. 扩展性自检

### Q1：新增一种渲染器（比如 `blockquote`）需要改哪些地方？

- 新增 `md/blockquote.go`：实现 Segment 接口
- 在 Registry 的 Detector 列表里注册一个 `BlockquoteDetector`
- 加配置项 `mdblockquote`

**不需要改**：Pipeline、Registry、Mapper、Inline Renderer、Host Integration。✅

### Q2：关闭一个渲染器（`mdtable: false`）后行为？

- Registry 跑 Detector 时跳过被关闭的
- 该类行落到下个 Detector（最终落到 paragraph 兜底）
- 用户看到的就是原文 fallback

✅ 配置可组合、行为可预期。

### Q3：`.md` 文件 vs 非 `.md` 文件的分流？

- Host Integration 层第一行就判断文件类型
- 非 `.md` → 我们的代码完全静默，Micro 原生显示

✅ 边界清晰，不会污染非 markdown 编辑体验。

## 9. 待讨论的开放问题

以下是设计过程中识别出来但未定论的问题，逐个讨论：

### O1：segment 缓存的失效粒度

行级 / 段级 / 全文？

- **全文最简单**：buffer 任何变更 → 下一帧重建所有 segment。但大文件性能差。
- **段级**：buffer 变更 → 只重建覆盖变更行的 segment。理想但实现复杂。
- **行级**：折中。

建议先做全文级，性能不行再优化。

### O2：多光标落在同一个 segment 内 vs 不同 segment 的处理差异

- 同一个 segment 内：该 segment 一个 edit state 标记即可
- 不同 segment：每个被命中的 segment 都标记 edit state
- 这两种 case 是否需要架构层面的特殊设计，还是 segment 自然处理？

倾向：segment 自然处理（`IsEditState(cursors)` 接收光标列表，自己遍历）。

### O3：渲染输出的中间表示用 `[]RenderCell` 还是更紧凑的 `([]byte, []tcell.Style)`？

- **`[]RenderCell`**：每个字符一个结构体，简单直观，但内存占用大（每字符 ~24 字节）
- **`([]byte, []tcell.Style)`**：两个并行 slice，紧凑（同色 run 共享 style 对象），但需要保持两个 slice 长度同步
- **混合**：用 `[]RenderCell` 但 run-length encode（连续相同 style 的字符压缩）

建议：v0.1 用 `[]RenderCell` 求简单，性能瓶颈出现再优化。

### O4：鼠标点击反向映射放在哪一层（Pipeline / Mapper / Segment）？

- **Pipeline**：每帧维护一份"屏幕 cell → buffer 位置"的反向映射表
- **Mapper**：作为 mapper 的另一面职责（mapper 本来就是双向的）
- **Segment**：每个 segment 自己回答"我的第 N 个输出字符对应 buffer 哪个位置"

建议：Mapper + Segment 协作。Segment 提供 `MapScreenToBuffer(screenOffset int) (buffer.Loc, bool)`，Mapper 在更高层做屏幕行 ↔ segment 输出行 的转换。

但这个议题不紧急——v0.5 才用到，可以先不定。

### O5：Registry 何时重建 segment？

- 每帧都跑 Detector？
- buffer 变更后跑 Detector？
- 视口变化后跑 Detector？

理想是 buffer 变更后增量重建。但需要 buffer 提供变更通知。

v0.1 简化：每帧跑一次 Detector，但只跑视口范围内的行。性能不够再加缓存。

### O6：Micro 原生渲染作为 edit-state fallback 时，要不要"完全跳过"我们的代码？

- 完全跳过 → 走 Micro 原来的 displayBuffer 路径
- 部分跳过 → 走我们的 displayBuffer 但 segment.Render() 直接调 Micro 原生逐 rune 逻辑

建议：**部分跳过**。让 displayBuffer 的"外层壳"始终是我们的代码（处理行号移除、状态栏等），只是 edit state 行的内容生成委托给 Micro 原生逐 rune 函数。这样外层逻辑（gutter、状态栏、视口控制）只写一份。

## 10. 与 PRD 第三章（双模式）的关系

本架构**与是否启用 PRD 第三章的双模式无关**：

- 探照灯模型（v0.1 起的默认行为）下：架构如本文档描述，光标驱动 segment edit state
- 双模式（v0.6+ 可选启用）下：在 Layer 4 (Pipeline) 之上**再加一层 Mode Controller**，控制"是否隐藏光标 / 是否拦截方向键 / 10 秒计时器"

```
┌─────────────────────────────────────────┐
│  Mode Controller（v0.6 才加入）          │
│  - 双模式状态机                          │
│  - 10 秒计时器                           │
│  - 输入事件劫持                          │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│  Layer 4: Pipeline                       │
│  （探照灯模型始终用这一层）              │
└─────────────────────────────────────────┘
```

**关键**：Mode Controller 是**可选的、可移除的、对下层透明**的。即使最终决定不做双模式，Pipeline 及以下的所有层都不受影响。
